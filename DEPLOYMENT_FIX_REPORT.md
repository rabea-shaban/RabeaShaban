# DEPLOYMENT_FIX_REPORT.md

## 1. Root Cause

The 404-on-refresh was caused by **three compounding issues** in the Vercel deployment configuration. All three had to be present simultaneously to produce the symptom.

### Primary cause â€” `vercel.json` used the deprecated `routes` API with `{ "handle": "filesystem" }`

The `vercel.json` on the `main` branch contained:

```json
{
  "cleanUrls": true,
  "routes": [
    { "handle": "filesystem" },
    { "src": "/(.*)", "dest": "/index.html" }
  ]
}
```

The `routes` key is the **legacy Vercel routing API**. When `{ "handle": "filesystem" }` is present, Vercel checks whether a real file exists at the requested path first. If no file is found, it moves to the next route. Since `/about`, `/projects`, `/services`, `/contact`, and `/certificates` are not real files in `dist/`, Vercel found no match and returned its own `404: NOT_FOUND` page â€” the fallback `dest: /index.html` was never reached.

The `routes` key is also **mutually exclusive** with the modern `rewrites`, `redirects`, and `headers` keys. Mixing them causes undefined behavior.

### Secondary cause â€” `public/vercel.json` shadowed the root `vercel.json`

A separate commit placed a `vercel.json` inside `public/`. Vite copies everything in `public/` verbatim into `dist/` at build time. Vercel reads deployment configuration from the output directory (`dist/`), so `dist/vercel.json` (sourced from `public/vercel.json`) was the config Vercel actually used on every deployment. The root `vercel.json` was completely ignored.

This meant that even if the root `vercel.json` was corrected, the broken `public/vercel.json` would always win after the next build.

### Tertiary cause â€” Route case mismatch: `/Certificates` vs `/certificates`

`App.jsx` defined the route as `path="/Certificates"` (capital C). Vercel's infrastructure runs on Linux, where the filesystem and URL routing are case-sensitive. Any request to `/certificates` (lowercase) â€” including links from the sitemap, search engines, or bookmarks â€” would fall through to the `*` wildcard route and render the NotFound page instead of the Certificates page.

## 3. Why the Issue Happened

The project went through several attempted fixes that each introduced a new problem:

1. **Commit `4dbe44f`** â€” Added `{ "handle": "filesystem" }` to `vercel.json` to try to serve `sitemap.xml` and other static files correctly. This broke SPA routing because the filesystem handler consumed all requests that didn't match a real file and returned 404 before the fallback could fire.

2. **Commit `ce76bb0`** â€” Placed a `vercel.json` inside `public/` to try to apply routing rules to the static output folder. This caused Vite to copy it into `dist/`, making it the authoritative config that Vercel read â€” permanently overriding the root `vercel.json` regardless of what it contained.

3. The `/Certificates` route was introduced with a capital C, which works on Windows (case-insensitive filesystem) and in local development, but fails on Vercel's Linux servers.

## 4. Why the Fix Works

### `vercel.json` â€” canonical SPA rewrite

```json
{
  "cleanUrls": true,
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/"
    }
  ]
}
```

The `rewrites` key is the **current Vercel SPA recommendation**. Unlike the legacy `routes` key, `rewrites` are applied **after** Vercel's static file serving layer. This means:

- Requests for real files (`/assets/index.js`, `/logo.png`, `/sitemap.xml`, `/robots.txt`, `/cv.pdf`) are served directly from `dist/` without ever reaching the rewrite rule.
- Requests for SPA routes (`/about`, `/projects`, `/services`, `/contact`, `/certificates`) find no matching file in `dist/`, so the rewrite rule fires and serves `/` â€” which resolves to `dist/index.html` via `cleanUrls: true`.
- React Router's `BrowserRouter` reads `window.location.pathname` on load and renders the correct page component.

Using `"destination": "/"` instead of `"destination": "/index.html"` is the canonical form in Vercel's current documentation. With `cleanUrls: true`, both are equivalent â€” Vercel resolves `/` to `index.html` automatically.

### Deleting `public/vercel.json`

With this file removed, Vite no longer copies any `vercel.json` into `dist/`. Vercel reads the root `vercel.json` directly from the repository root, which is the correct and intended behavior.

### Lowercase `/certificates` route

Aligning the route path, sitemap, and all internal links to lowercase `/certificates` ensures consistent behavior across all environments and prevents case-sensitivity 404s on Linux servers.

## 5. References â€” Latest Vercel SPA Deployment Recommendations

The official Vercel documentation for deploying Single Page Applications recommends:

**`vercel.json`:**
```json
{
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/"
    }
  ]
}
```

Key points from the official docs:
- Use `rewrites`, not the legacy `routes` key
- `rewrites` run after static file resolution â€” no need for negative lookahead regex to protect assets
- `"destination": "/"` is preferred over `"destination": "/index.html"` when `cleanUrls: true` is set
- Never place `vercel.json` inside `public/` â€” it will be copied into the output directory and shadow the root config
- The `routes` key cannot be combined with `rewrites`, `redirects`, or `headers`

**References:**
- https://vercel.com/docs/projects/project-configuration#rewrites
- https://vercel.com/docs/frameworks/vite
- https://vercel.com/docs/edge-network/rewrites
